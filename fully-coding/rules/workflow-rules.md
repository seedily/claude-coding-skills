# fully-coding 工作流规则

## 1. 串行执行规则（强制执行）

### 禁止 Agent 并行（红线）

- **步骤串行**：标准模式必须按 1→2→3→4→5→6→7→8→9 顺序执行；`--plan-only` 按 1→2→3→4 执行；`--quick-dev` 按最小需求记录→3→5→6→7 执行。任一模式内不可并行。
- **角色串行**：每个步骤只能激活一个主导角色
- **输入依赖**：步骤 N 开始前，步骤 N-1 的输出文档必须已落盘
- **禁止并行 Agent**：严禁使用以下方式：
  - TeamCreate 工具创建 Agent Team
  - Agent 工具 spawn 多个 general-purpose 或其他 subagent_type
  - Agent 工具的 `run_in_background` 参数
  - feature-split-team 或其他多 Agent 并行 Skill
  - **步骤 6 特例**：DBA 与代码评审员必须在同一响应中完成评审，由代码评审员合并输出最终评审结论，**不得**拆分为两个 Agent 先后 spawn 或并行运行
- **执行检查**：每步开始前 Orchestrator 必须验证：
  - 当前未运行任何 Agent Team
  - 当前未 spawn 任何子 Agent
  - 本步骤不会触发 TeamCreate
  - 若发现违规，立即阻断并向用户报告

## 2. 文档驱动规则

- **每步必盘**：每个步骤结束后，必须在 30 秒内将结果追加到 `.dev-log/{ts}-userStory.md` 或 `.dev-log/{ts}-codingLog.md`
- **文档即接口**：下游角色只读取上游文档，不读取上游角色的“记忆”
- **模板遵循**：userStory.md 和 codingLog.md 必须严格使用 `templates/` 中的模板格式
- **时间戳一致**：同一次任务的所有文档使用同一时间戳 `{yyyyMMdd-HHmmss}`

## 3. 步骤输入/输出契约

### 3.0 执行模式契约

| 模式 | 参数 | 执行范围 | 完成状态 | 约束 |
|---|---|---|---|---|
| 标准模式 | 无 | Step 1-9 | Step 9 后 `status=completed` | 完整交付闭环 |
| 方案模式 | `--plan-only` | Step 1-4 | Step 4 后 `status=planned` | 不修改业务代码，不运行 Step 5-9 |
| 轻量开发模式 | `--quick-dev` | 最小需求记录 → Step 3 → Step 5 → Step 6 → Step 7 | Step 7 后 `status=completed` | 不执行 Step 4/8/9；仅记录必要需求、范围、实现、评审和测试 |

`--plan-only` 与 `--quick-dev` 不得同时使用；若同时出现，必须阻塞并要求用户选择一种模式。`--plan-only` 产物可通过后续 `/fully-coding --start-step 5 --task-id {ts}` 继续进入实现闭环。

步骤契约定义每一步的最低完成标准，用于 Orchestrator 判断能否进入步骤、能否离开步骤、断点续跑时应从哪里恢复。契约只约束输入/输出与完成标准，不替代各角色文档中的执行细节。

### 3.1 通用契约

所有步骤必须遵守：

- **进入前检查**：确认本步骤进入条件满足；当且仅当**同时**传入 `--start-step 3` 且 `--batch-subtask` 的 batch 子任务从步骤 3 开始并豁免"步骤 N-1 输出文档已落盘"检查（但仍须满足本步骤必填输入，且子任务 `userStory.md` 已由 batch Orchestrator 在拆分阶段填写完整，见 `../../fully-coding-batch/rules/batch-workflow-rules.md §2`）；其余情况下步骤 N 开始前必须确认步骤 N-1 输出文档已落盘。
- **执行中上报**：执行期间必须按 `heartbeat-resume-rules.md` 更新 `last-heartbeat`、`current-step`、`current-role`、`pid`。
- **离开前检查**：确认本步骤必填输出字段已写入对应文档；若退出条件不满足，不得进入下一步。
- **续跑判断**：若章节标题存在但必填输出为空，视为该步骤未完成，续跑时应从该步骤继续。
- **阻塞处理**：若无法补齐必填输出且流程无法继续，必须按第 8 节「异常处理规则」登记阻塞。

### 3.2 Step 1：需求检索

**进入条件**：用户原始提示词存在；已读取 `project-config.md`；已生成任务时间戳 `{ts}`；已创建或准备创建 `{ts}-userStory.md`。`--quick-dev` 模式可只写最小需求记录，但必须保留原始提示词和特性摘要。

**必填输入**：用户原始提示词；`{{config.knowledge_dir}}/{{config.requirement_doc}}`，若不存在需记录说明。

**必填输出**：写入 `{ts}-userStory.md` 的「原始提示词」「特性摘要」「需求终稿命中情况」「相关章节」「与本次需求的差异」。

**退出条件**：`userStory.md` 已存在；「原始提示词」「特性摘要」「需求检索结果」非空；若需求文档缺失，已记录缺失说明和替代判断依据。

### 3.3 Step 2：需求生成

**进入条件**：Step 1 退出条件已满足；`userStory.md` 已存在。

**必填输入**：Step 1 的需求检索结果；用户原始提示词；已有需求终稿命中情况。

**必填输出**：写入 `{ts}-userStory.md` 的「新增/变更需求」「验收标准」「需求假设」「需求歧义记录」「关联文档」。

**退出条件**：「新增/变更需求」至少包含 1 条需求条目；「验收标准」至少包含 1 条 AC；若需求存在歧义但可继续，已记录假设；若需求矛盾且无法判断，登记阻塞。`--quick-dev` 模式可跳过完整 Step 2，但必须在 `userStory.md` 写入最小需求记录和关键验收点后才能进入 Step 3。

### 3.4 Step 3：开发范围

**进入条件**：普通任务必须满足 Step 2 退出条件；`{ts}-codingLog.md` 已存在或可创建。batch 子任务当且仅当同时传入 `--start-step 3` 与 `--batch-subtask` 时可从 Step 3 开始并豁免 Step 2 输出落盘检查，但必须已有由 batch Orchestrator 在拆分阶段填写完整的 `userStory.md`。

**必填输入**：`{ts}-userStory.md`；技术方案文档；服务设计文档；错误码文档。

**必填输出**：写入 `{ts}-codingLog.md` 的「影响服务」「影响模块」「改动文件清单」「git 分支信息」「删除/废弃类变更检查清单（如适用）」；**「全栈覆盖声明」**——按 `implementation-checklist-rules.md §0 全栈覆盖检查` 的表格格式，逐一声明本功能涉及哪些层（后端Controller/Service/Repository/PO/前端页面/前端API/第三方集成/数据库），每层的实现内容和完成状态。

**退出条件**：「影响服务」「影响模块」「改动文件清单」已填写；**「全栈覆盖声明」已填写，覆盖后端/前端/数据/第三方集成全部可能受影响的层，每层注明实现内容**；若暂时无法定位具体文件，必须写明原因和下一步定位方式；若涉及删除/废弃类变更，必须完成残留引用检查项；**若涉及前端页面/菜单/路由，步骤 3 必须记录「前端目录约定」，说明现有 `src/views/` 目录风格及本次新增页面实际路径**。

### 3.5 Step 4：开发方案

**进入条件**：Step 3 退出条件已满足；已确定影响模块和改动范围。

**必填输入**：`userStory.md`；`codingLog.md` Step 3；功能实现模板；功能实现概览；DDD / 编码 / 持久层 / 安全 / 异常规则；错误码文档。

**必填输出**：写入 `{ts}-codingLog.md` 的「方案摘要」「新增/更新的功能实现文档路径」「文档处理方式与原因」「关键设计决策」「错误码使用情况」「幂等性设计说明」。

**退出条件**：方案能直接指导开发者编码；接口、数据模型、核心逻辑、异常处理已说明；写操作已说明幂等性策略，或说明为什么不需要幂等；若新增错误码，已记录取值、含义、使用场景；**若涉及前端页面，方案中的前端文件路径必须与步骤 3 记录的现有 `src/views/` 目录风格一致，否则在「代码与文档偏差说明」中提前记录**。`--plan-only` 模式下，Step 4 完成后更新 `codingLog.md` frontmatter：`status=planned`，然后停止流程并向用户报告方案产物路径。

### 3.6 Step 5：开发实现

**进入条件**：Step 4 退出条件已满足；开发方案中涉及的文件路径或模块边界已明确。`--quick-dev` 模式不执行 Step 4，可在 Step 3 退出后直接进入，但必须先在 `codingLog.md` Step 5 记录轻量实现思路，且 Step 5 必填输入中的「codingLog Step 4」对 quick-dev 不适用（改为读取 Step 3 方案要点 + userStory）。

**必填输入**：`codingLog.md` Step 4；相关规则文件；现有代码；**`implementation-checklist-rules.md`（全栈实现 Checklist）**。

**必填输出**：写入 `{ts}-codingLog.md` 的「代码变更」「编译结果」「错误码使用情况」「幂等性实现情况」「删除/废弃类残留引用检查结果（如适用）」；**「全栈覆盖确认」**——对照 Step 3 的「全栈覆盖声明」，逐项确认各层是否已完成实现，若某层标记为涉及但未完成，必须说明原因。

**退出条件**：代码实现完成；编译或基础校验已执行，若无法执行必须写明原因；新增写接口已实现幂等性或说明替代方案；新增错误码已记录，供 Step 8 更新文档；**「全栈覆盖确认」中所有标记为"涉及"的层均已完成实现，或已完成但未完成的层已有明确的阻塞记录和原因说明**；不得带明显编译错误进入 Step 6。

### 3.7 Step 6：代码评审

**进入条件**：Step 5 退出条件已满足；代码变更清单已记录；编译结果已记录或有无法执行说明。

**必填输入**：本次代码 diff；`codingLog.md` Step 5；所有 rules；错误码文档。

**必填输出**：写入 `{ts}-codingLog.md` 的「DBA 意见」「代码评审员意见」「问题清单」「问题分级」「修复结果」「评审结论」。

**退出条件**：所有问题均已分类；auto-fix 问题已修复或记录保留原因；block 问题仅在达到阻塞标准时登记；幂等性、错误码、SQL/缓存、安全规则已检查；评审结论为「通过」或「需自动修复」才可进入 Step 7（「需自动修复」指已进入或即将进入修复循环，修复循环结束、问题清零或可接受后即满足进入 Step 7 条件）。`--quick-dev` 模式下，Step 6 评审通过后直接进入 Step 7，Step 7 通过后即可完成任务。

### 3.8 Step 7：测试用例

**进入条件**：Step 6 评审结论为「通过」或「需自动修复」且自动修复循环已结束（问题清零或可接受）。

**必填输入**：已评审通过的代码；`codingLog.md` Step 6；相关测试规范或已有测试。

**必填输出**：写入 `{ts}-codingLog.md` 的「测试覆盖」「测试结果」「未覆盖项说明」。

**退出条件**：已补充或更新必要测试；测试已执行，若无法执行必须说明原因；失败测试已自动修复，或达到阻塞条件后登记 blockLog；不得带未解释的失败测试进入 Step 8。`--quick-dev` 模式下，Step 7 通过后将 `codingLog.md` frontmatter 更新为 `status=completed`，并在 Step 7 记录轻量模式完成说明。

### 3.9 Step 8：文档更新

**进入条件**：Step 7 退出条件已满足；代码变更、测试结果、错误码使用情况均已记录。

**必填输入**：`codingLog.md` Step 3-7；最终代码实现；技术方案文档；服务设计文档；功能实现文档；错误码文档。

**必填输出**：写入 `{ts}-codingLog.md` 的「更新文档清单」「错误码文档更新」「代码与文档偏差说明」「公共资源变更摘要」。

**退出条件**：涉及的知识库文档已同步；新增错误码已反向更新错误码文档；新增/修改 `{{config.feature_impl_glob}}` 时，已同步 `{{config.knowledge_dir}}/{{config.feature_impl_overview}}`；涉及菜单、页面、路由或权限入口变更时，已同步 `{{config.knowledge_dir}}/{{config.feature_impl_menu_iteration}}`；**涉及前端页面时，必须在 `codingLog.md`「代码与文档偏差说明」中核对并记录实际新增/修改页面路径与步骤 3/4 计划路径的一致性**；**涉及模块结构/分层/网关/领域服务职责变更时，已更新 `{{config.knowledge_dir}}/{{config.tech_architecture_doc}}`（见 §7.4 触发矩阵）**；若 batch 子任务修改公共资源，已记录到公共资源变更摘要。

### 3.10 Step 9：自主进化

**进入条件**：Step 8 退出条件已满足；`codingLog.md` frontmatter `status != blocked`。

**必填输入**：`userStory.md`；`codingLog.md`；本次文档更新结果。

**必填输出**：`{ts}-suggestion.md`；`codingLog.md` Step 9；`codingLog.md` 任务完成状态。

**退出条件**：已完成文档完整性自检；已生成 suggestion.md；已更新 `codingLog.md` frontmatter：`status=completed`；已追加任务完成时间和产出清单。

## 4. 规范引用规则

- 步骤 3/4/8（架构师）按 `rules/runtime-load-map.md` 先读取本步骤必读文件；设计涉及分层、持久层、安全或 Git 时，再读取并引用 `rules/backend-ddd-layer-rules.md`、`rules/persistence-sql-cache-rules.md`、`rules/security-rules.md`、`rules/git-rules.md` 和 `{{config.knowledge_dir}}/{{config.error_code_doc}}`
- 步骤 5/7（开发者）按 `rules/runtime-load-map.md` 读取开发必需规范；实现必须符合 DDD、编码、持久层、安全、异常、Git 与错误码规则，发现不确定细节时读取对应完整规则文件
- 步骤 6（代码评审员）默认以 `rules/review-checklist.md`、本次 diff、`{{config.knowledge_dir}}/{{config.error_code_doc}}` 作为评审入口；命中具体风险时再读取对应完整规则文件确认细节

## 5. 中断与确认规则

| 类别 | 操作清单 |
|------|----------|
| 自动执行 | 文档生成、代码编辑（≤10 文件）、本地测试、Grep/Read/Glob |
| 简要说明后执行 | 批量重构（>10 文件）、新增公共模块、修改接口契约 |
| 必须确认 | git push、不可逆 DDL、删除已有功能、修改 CI/CD、涉及密钥 |

**超量提示**：当单次任务代码编辑超过 30 个文件时，步骤 9 必须在 `{ts}-suggestion.md` 中生成「需求拆分建议」，提示用户下次将类似规模的需求拆分为多个小任务，以提升自动化执行效率与代码评审质量。

## 6. 代码质量规则与自动回归

### 基础质量规则
- 步骤 5 结束后代码必须能编译通过
- 步骤 6 评审意见必须引用具体文件路径和行号
- 步骤 7 测试失败必须修复，禁止带失败测试进入步骤 8

### 问题分级定义

评审发现的问题分为两个级别，**只有「阻塞」级别才会中断流程**：

#### 自动修复（auto-fix）—— 默认级别

以下类型的问题**一律自动修复**，不中断流程、不等待用户确认：

| 类别 | 示例 |
|------|------|
| 代码规范 | 命名不规范、缺少 Javadoc、魔法数字、代码风格 |
| 类型安全 | 使用 Map 代替 DTO、缺少泛型参数、类型转换不安全 |
| 业务校验缺失 | 缺少唯一性校验、缺少存在性校验、缺少边界条件检查 |
| 数据库问题 | 缺少索引、字段类型不合理、缺少默认值、TEXT vs JSONB |
| 前端缺陷 | 缺少 UI 功能（排序/搜索/分页）、表单验证不完整 |
| 序列化/格式 | 前后端数据格式不一致、JSON 解析方式不健壮 |
| 架构调整 | 职责划分不当、层级依赖方向错误 |
| 性能优化 | N+1 查询、循环调用、大对象未分页 |
| 测试不足 | 缺少关键路径测试、边界条件未覆盖 |

#### 阻塞（block）—— 仅限致命问题

**只有以下情况才可标记为「阻塞」，中断流程等待用户确认**：

| 类别 | 条件（必须同时满足） |
|------|---------------------|
| 安全漏洞 | 存在 SQL 注入、XSS、硬编码密钥等可被外部利用的漏洞，且修复涉及架构层面的重大变更 |
| 数据损坏风险 | 可能导致生产数据丢失或不可逆损坏，且修复方案需要用户决策（如 DDL 变更策略） |
| 接口契约破坏 | 修改了已发布的 API 契约，会导致其他服务调用方中断，且无法通过向后兼容方式修复 |
| 需求歧义 | 需求本身存在矛盾或重大遗漏，无法由 AI 判断正确方向，必须用户澄清 |

**关键原则**：
- 阻塞的判断标准极其严格：必须是**致命且无法自动修复**的问题
- 如果问题可以通过修改代码/SQL/配置来解决，即使改动量大，也应自动修复
- 如果不确定是否阻塞，**默认选择自动修复**
- 「改动量大」不构成阻塞理由

### 自动循环修复（最多 3 次）

#### 步骤 6 评审自动化（评审 → 自动修复 → 验证循环）

步骤 6 采用「评审 → 自动修复 → 验证」的闭环机制：

1. **评审阶段**（DBA + Reviewer，同一响应内分两段输出，**不得**拆成两个 Agent 并行或先后 spawn——红线见 §1）：
   - **DBA 段（先行、独立成段）**：聚焦表结构、索引、字段类型、SQL/缓存、迁移脚本、数据安全与幂等落库。**只要 Step 5 涉及 DDL / SQL / 索引 / 持久层变更，DBA 段为必填**，须给出结构审查结论并引用具体表/字段/索引与代码行号；无持久层变更时可只写一句"本变更未触及持久层，DBA 段无意见"，**不得与 Reviewer 段混写或整体省略**。
   - **Reviewer 段（随后）**：聚焦编码规范、类型安全、业务校验、序列化/格式、架构分层、性能、测试覆盖。
   - 两段统一按上述分级标准对每个问题标记 `auto-fix` 或 `block`。
   - 对 `auto-fix` 问题，**必须提供具体修复方案**（含修复代码片段或操作步骤）。
   - 对 `block` 问题，必须详细说明阻塞原因和可选方案。
   - 输出评审结论：「通过（无问题）」或「需自动修复（N 项）」或「阻塞（N 项）」。

2. **自动修复阶段**（Developer）：
   - 若评审结论为「需自动修复」：
     - Orchestrator 自动切换到 Developer 角色
     - Developer 按评审提供的修复方案逐项修复
     - 修复完成后执行编译验证
     - **不登记 blockLog.md**（流程可继续，无需用户确认）
   - 若评审结论为「阻塞」：
     - 仅在 3 次自动修复循环后仍有无法解决的阻塞问题时才触发
     - 向用户报告阻塞原因和修复建议
     - **登记到 `.dev-log/{ts}-blockLog.md`**（按模板格式追加阻塞事件，标记「待确认」）
     - 暂停流程，等待用户决策

3. **验证阶段**（DBA + Reviewer）：
   - 修复后重新评审，确认修复是否到位
   - 若仍有 `auto-fix` 问题，继续修复（计入循环次数）
   - 若全部问题已修复或仅剩可选优化项，评审通过，进入步骤 7

4. **循环控制**：
   - 「评审 → 自动修复 → 验证」子循环最多 **3 次**
   - 循环修复期间**不登记 blockLog.md**
   - 3 次后仍有 `auto-fix` 问题未解决：由 Orchestrator 评估是否可接受（多数情况下可接受并继续），记录到 codingLog.md 但不阻塞
   - 3 次后仍有 `block` 问题未解决：**登记 blockLog.md，暂停流程**

#### 步骤 7 测试自动化
- 测试失败后：
  - 分析失败日志，定位失败原因
  - 尝试自动修复代码（如参数错误、逻辑缺陷）
  - 重新运行测试，最多循环 3 次
  - **循环修复期间不登记 blockLog.md**
- 3 次后仍失败（流程无法继续，必须用户确认）：
  - 向用户报告失败测试清单和错误详情
  - **登记到 `.dev-log/{ts}-blockLog.md`**（按模板格式追加阻塞事件，标记「待确认」）
  - 暂停流程，等待用户决策

### 循环次数记录

每次循环修复必须在 codingLog.md 中记录：

```markdown
## 步骤 6：代码评审（第 N 次循环）

- 循环次数：N/3
- 评审问题：...（标记 auto-fix / block）
- 自动修复：...（列出已修复项）
- 验证结果：...（通过 / 仍有问题）
- 评审结论：...
```

## 8. 异常处理规则

### 阻塞登记（强制）

**仅在流程中断、必须暂停等待用户确认时**，才执行 blockLog.md 登记。

以下情况**不登记** blockLog.md（但需记录到 codingLog.md 或 userStory.md 对应章节）：
- 步骤 6 评审「阻塞」但尚在 3 次循环修复期内
- 步骤 7 测试失败但尚在 3 次循环修复期内
- 步骤 6 评审结论为「需自动修复」且尚在 3 次循环修复期内（流程可自动继续）

以下情况**必须登记** blockLog.md：
- 循环修复达到上限（3 次）后仍无法解决
- 某步骤执行失败且当前角色无法自行修复
- 需要用户做破坏性/不可逆决策（如删除已有功能、修改 CI/CD）

**登记动作**：

1. **创建/追加 blockLog.md**：
   - 若 `.dev-log/{ts}-blockLog.md` 不存在，按 `templates/block-log-template.md` 创建
   - 追加新的阻塞事件记录，标记「用户确认状态：待确认」
2. **登记流程文档**：
   - 步骤 1-2 的阻塞：追加到 `.dev-log/{ts}-userStory.md` 的「阻塞记录」章节
   - 步骤 3-8 的阻塞：追加到 `.dev-log/{ts}-codingLog.md` 对应步骤章节
3. **向用户报告**：阻塞步骤、阻塞原因、修复建议、所需决策

### 用户修复后的 blockLog.md 更新

用户确认已处理阻塞后，**必须**更新 blockLog.md：

1. 找到该阻塞事件，将「用户确认状态」改为「已确认」
2. 在「用户处理方案」字段记录用户的处理方案或决策内容
3. 在「解决时间」字段填写当前时间
4. 在「解决方式」字段选择：用户决策 / 人工修复 / 放弃任务

### auto-resume 与 blockLog.md 联动

自动续跑时，**必须先检查 blockLog.md**：

1. 若存在 blockLog.md 且包含「用户确认状态：待确认」的阻塞事件：
   - **不得直接恢复执行**
   - 向用户报告该未解决阻塞：阻塞步骤、原因、修复建议
   - 询问用户是否已处理（Y/n）
   - 用户确认已处理：执行「用户修复后的 blockLog.md 更新」，然后继续恢复
   - 用户未处理：保持暂停，本轮 resume 跳过
2. 若 blockLog.md 不存在或所有阻塞事件均已确认：
   - 按正常 auto-resume 逻辑继续

### 一般异常处理

- 若某步骤执行失败，当前角色应尝试一次自我修复
- 若无法自行修复，执行上述「阻塞登记（强制）」后暂停流程，等待用户决策
- 不得在没有用户确认的情况下执行 git push、force push、rm -rf 等危险操作
- 若发现 Agent 并行违规行为，立即阻断流程，记录违规类型（TeamCreate/spawn/background）到 blockLog.md 和 codingLog.md，并向用户报告

## 9. 外部依赖约束

- **允许 git-sync**：步骤 8 完成后，对于有变更的代码工程，若其内部有初始化 git 配置（且 git remote 有声明代码仓库），则可对其执行 git add/commit/push（遵循 `rules/git-rules.md` 规范）
- **禁止自动推送**：不得在步骤 1-7 中自动执行 git push，所有推送必须在步骤 8 完成之后，且需要用户确认，若用户在1分钟内没有响应，可以完结本次任务，并提示用户做后续的操作（如 git push）
